# Git and Branching Strategy Rules for Agents

## Russian (Правила ветвления и коммитов):
1. **Создание ветки:** Перед началом любых изменений в коде существующего проекта, агент ОБЯЗАН создать новую ветку (Git branch). Никакие изменения не должны производиться напрямую в ветке `main`.
2. **Пуш изменений:** После внесения изменений агент должен закоммитить и запушить рабочую ветку в удаленный репозиторий GitHub.
3. **Пул-реквест:** По окончании работы над задачей агент должен создать новый Pull Request (пул-реквест) для проверки (review) и ожидать подтверждения пользователя перед слиянием (merge) изменений с основной веткой `main`.

## English (Branching and Commit Rules):
1. **Branch Creation:** Before starting any modifications or coding task on the existing codebase, the agent MUST create a new Git branch. Direct modifications on the `main` branch are strictly prohibited.
2. **Pushing Changes:** After making changes, the agent must commit and push the active branch to the remote GitHub repository.
3. **Pull Request:** Upon completion of the task, the agent must create a new Pull Request (PR) for review and wait for explicit user approval before merging any changes back into the `main` branch.
